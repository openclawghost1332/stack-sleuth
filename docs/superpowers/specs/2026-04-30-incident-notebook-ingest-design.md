# Design: Stack Sleuth Incident Notebook Ingest

## Problem Statement

Stack Sleuth already has a strong deterministic analysis ladder, but it still expects operators to manually rewrite real incident notes into strict `@@` incident packs or `@@@` portfolios before the existing engines can help. Real handoff artifacts are more often markdown notebooks, issue drafts, or incident notes with headings, prose, and fenced trace blocks. Notebook ingest should remove that format friction without creating a parallel analysis stack.

## What Makes This Cool

Paste a markdown incident handoff note and get the same Incident Pack Briefing or Portfolio Radar result without manually translating it first. That makes Stack Sleuth feel less like a parser collection and more like a portable incident workbench.

## Premises

1. The bottleneck is input-shape friction, not missing analysis engines.
2. A deterministic markdown-heading parser is enough for a compelling first slice.
3. Notebook ingest should normalize into existing pack and portfolio formats instead of inventing a notebook-only analyzer.

## Approaches Considered

### 1. Minimal viable: CLI-only notebook import
- Summary: parse markdown headings into one normalized incident pack and expose only a CLI flag.
- Effort: S
- Risk: Low
- Pros: fastest ship, smallest surface area.
- Cons: less visible, weaker outsider demo.
- Reuses: existing pack and portfolio engines.

### 2. Recommended: shared notebook parser plus CLI and browser routing
- Summary: add one shared parser that recognizes markdown notebook headings and routes normalized output into Incident Pack Briefing or Portfolio Radar.
- Effort: M
- Risk: Medium
- Pros: stronger product story, visibly compounds the repo, keeps the implementation deterministic.
- Cons: more integration work, more test coverage needed.
- Reuses: existing pack and portfolio analyzers, shared renderers, browser cards, and copy flows.

### 3. Overreach: notebook import plus source-specific adapters
- Summary: add Slack, GitHub, Jira, or transcript-specific parsing and metadata extraction.
- Effort: XL
- Risk: High
- Pros: bigger wow factor.
- Cons: brittle and too wide for one cycle.
- Reuses: existing engines, but adds too much format variance.

## Recommended Approach

Choose approach 2. It is the boldest move with a real one-cycle path and directly compounds Stack Sleuth into a thicker workflow product.

## Accepted Scope

- Shared markdown notebook parser that recognizes heading aliases for current, history, baseline, candidate, timeline, and optional `# Pack:` labels
- Normalization into existing `@@` incident-pack and `@@@` portfolio text shapes
- CLI notebook workflow with text, markdown, and JSON output
- Browser auto-detection, example data, and copy behavior for notebook input
- README and tests covering pack-style and portfolio-style notebook ingest

## Deferred

- Slack, GitHub, Jira, and chat-specific adapters
- Rich metadata extraction beyond headings and labels
- Persistent storage, uploads, attachments, or collaboration
- AI-generated remediations or notebook cleanup

## Architecture

Add `src/notebook.js` as the single normalization layer. It parses markdown headings, resolves supported aliases to canonical pack sections, optionally groups multiple `# Pack:` blocks into a portfolio, and emits normalized text. The CLI and browser should route notebook input into the already-existing Incident Pack Briefing or Portfolio Radar engines and reuse the same summaries and serializers.

## Components

- `src/notebook.js`: notebook parsing, canonical section mapping, normalization text, routing helper
- `bin/stack-sleuth.js`: `--notebook` entrypoint, validation, serializer wrappers
- `src/main.js`: notebook auto-detection, browser render path, clipboard copy path
- `src/examples.js`, `README.md`, `index.html`: outsider-facing workflow discoverability
- `tests/notebook.test.mjs`, `tests/cli.test.mjs`, `tests/browser-copy.test.mjs`, `tests/examples.test.mjs`, `tests/readme.test.mjs`: behavior coverage

## Data Flow

1. User pastes a markdown incident note or grouped `# Pack:` notebook.
2. `parseIncidentNotebook` normalizes headings into canonical pack sections.
3. Notebook output becomes either normalized incident-pack text or normalized portfolio text.
4. Existing pack or portfolio analyzers run unchanged.
5. Existing renderers present summaries in text, markdown, JSON, or browser cards.

## Error Handling

- Empty notebook input returns an unsupported notebook result.
- Notebook headings with no supported sections return a clear reason.
- Normalized pack notebooks that still produce no runnable analyses fail loudly in CLI mode.
- Normalized portfolios with no runnable packs fail loudly in CLI mode.

## Testing

- Unit tests for notebook parsing, alias recognition, pack grouping, normalization, and routing
- CLI tests for `--notebook` success, serialization, and validation failures
- Browser tests for notebook detection, rendering, example loading, and copy behavior
- README and examples tests to keep the public workflow visible

## Success Criteria

- A markdown incident note can be pasted directly into Stack Sleuth and routed into a valid briefing or portfolio result.
- The feature reuses existing analyzers instead of duplicating them.
- The notebook workflow is legible to a stranger from the README and browser UI in under 30 seconds.
