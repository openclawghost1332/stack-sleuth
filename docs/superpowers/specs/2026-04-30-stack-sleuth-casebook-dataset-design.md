# Stack Sleuth Casebook Dataset Design

Generated by autonomous builder-mode scoping on 2026-04-30.
Status: DRAFT

## Problem Statement
Stack Sleuth can rank a labeled incident portfolio, forge reusable casebook exports, and merge a living casebook, but it still does not produce a first-class portable dataset that another person or future run can save and reuse directly. The next compounding step is to turn release triage into a durable incident-memory artifact.

## What Makes This Cool
One portfolio can now produce both human-readable triage output and a portable machine-readable dataset. That dataset can travel between terminals, docs, and future Stack Sleuth sessions without manual rewriting.

## Premises
1. Reusable incident memory is a better next step than another isolated analysis mode.
2. The narrowest strong slice is a deterministic dataset built on top of current Portfolio Radar and Casebook Merge outputs.
3. Re-ingest should reuse the existing `--history` Casebook Radar path instead of inventing a second lookup command.

## Approaches Considered

### Approach A: Lean on existing merge JSON
Summary: Tell users to save `--merge-casebook --json` output.
Effort: S
Risk: Low
Pros: Tiny diff, no new workflow to learn.
Cons: Weak contract, misses response queue and recurring portfolio context, thin public story.
Reuses: Current merge JSON.

### Approach B: Dedicated Casebook Dataset workflow
Summary: Add a shared dataset builder on top of portfolio plus merge outputs, expose it through CLI and browser, and let Casebook Radar reuse the saved dataset through `--history`.
Effort: M
Risk: Medium
Pros: Clear reusable artifact, strong demo surface, direct future reuse.
Cons: More plumbing across CLI, browser, tests, and docs.
Reuses: `analyzeIncidentPortfolio`, `analyzeCasebookMerge`, existing casebook export text.

### Approach C: Persistent dataset management
Summary: Add saved datasets, editing, import flows, and conflict review state.
Effort: XL
Risk: High
Pros: Strongest long-term product.
Cons: Too much lifecycle and UI work for one cycle.
Reuses: All current casebook workflows.

## Recommended Approach
Approach B. It creates a durable product boundary, compounds future runs, and stays feasible inside one aggressive cycle.

## Architecture
- Add `src/dataset.js` as the shared Casebook Dataset engine.
- The dataset builder accepts either a raw portfolio input or an already analyzed portfolio report.
- The dataset payload includes a stable kind/version marker, portfolio summary, response queue, recurring incident and hotspot summaries, merged cases, and the reusable labeled export text.
- `bin/stack-sleuth.js` adds `--dataset <path|->` and routes text, markdown, or JSON output through shared renderers.
- Casebook Radar history loading first checks whether the input is a Stack Sleuth dataset JSON blob. If yes, it converts the saved export text into labeled history batches before calling the current casebook analyzer.
- The browser portfolio flow renders dataset summary plus dataset export text in a dedicated card and keeps Portfolio Radar as the primary result.

## Error Handling
- `--dataset` fails when the input is not a labeled portfolio or contains no runnable packs.
- Dataset re-ingest through `--history` fails with a clear message when the JSON kind is unrecognized or the saved dataset has no usable export text.
- Non-dataset history input continues down the current labeled-history path unchanged.

## Testing
- New dataset unit tests cover building the dataset from a portfolio report and validating re-ingest parsing.
- CLI tests cover `--dataset` text/JSON/markdown output plus `--history` consuming a saved dataset file.
- Browser tests cover portfolio rendering of the dataset card and reset behavior when leaving portfolio mode.
- README assertions update so docs stay aligned with the shipped workflow.

## Success Criteria
1. `stack-sleuth --dataset <portfolio>` emits a stable dataset summary and reusable export.
2. `stack-sleuth --dataset <portfolio> --json > history.json` can be fed back into `stack-sleuth --history history.json`.
3. The browser portfolio flow visibly shows the new dataset card without displacing Portfolio Radar as the main result.
4. Tests and docs cover the new workflow.

## Deferred
- Persistent local storage.
- Conflict editing or guided merge resolution.
- Arbitrary external schema import.
