# Design: Stack Sleuth Casebook Forge

Generated by office-hours + brainstorming on 2026-04-30
Status: APPROVED
Mode: Builder
Operator mode: Autonomous evolution cycle, using the cycle brief as implicit approval to proceed.

## Problem Statement
Stack Sleuth can already analyze a current batch, compare two releases, inspect a rollout timeline, compose a single incident pack briefing, and rank several packs with Portfolio Radar. What it still cannot do is turn that portfolio into a reusable incident-memory asset. Teams still have to hand-curate `=== label ===` history files after triage, which breaks the compounding loop.

## What Makes This Cool
Paste a release-worth of incident packs, get a ranked queue and a forged casebook you can save immediately. That makes Stack Sleuth feel less like a one-off parser and more like a real incident-memory engine: the same investigation input now produces a reusable dataset for the next incident.

## Goal With This Cycle
Ship the narrowest version of “portfolio in, reusable casebook out” that feels obviously useful to an outsider and compounds future Stack Sleuth workflows.

## Premises
1. The highest-leverage next step is to make Portfolio Radar produce reusable memory, not just prioritization.
2. A deterministic export that reuses existing digest, casebook, and portfolio signals is better than inventing a new storage system this cycle.
3. The output must be directly usable in later Stack Sleuth workflows, especially Casebook Radar and Incident Pack Briefing history sections.
4. The browser demo should surface the forged casebook, but the reusable core belongs in a shared engine and CLI workflow first.

## Approaches Considered

### Approach A: Portfolio summary plus copy-only appendix
- Summary: Append a lightly formatted export block to Portfolio Radar and let users manually save it.
- Effort: S
- Risk: Low
- Pros: Fastest ship, smallest diff, easy to explain.
- Cons: Weak product boundary, muddles portfolio prioritization with dataset generation, thin JSON story.
- Reuses: `src/portfolio.js`, existing copy flows.

### Approach B: Dedicated Casebook Forge workflow on top of portfolio input
- Summary: Add a shared forge engine that ingests a labeled incident portfolio, deduplicates signatures, suggests reusable case labels, tracks source packs and matched history labels, and emits a ready-to-save `=== label ===` casebook export plus text/markdown summaries.
- Effort: M
- Risk: Low-Med
- Pros: Real reusable engine, clear CLI/browser workflow, directly compounds future Casebook Radar runs.
- Cons: Requires a new module, new UI cards, and careful copy behavior.
- Reuses: `src/portfolio.js`, `src/briefing.js`, `src/casebook.js`, `src/digest.js`, `src/analyze.js`.

### Approach C: Persistent incident-memory studio
- Summary: Add editing, local persistence, import/export, and manual curation workflows.
- Effort: XL
- Risk: High
- Pros: Biggest long-term surface area.
- Cons: Pushes the repo toward app-state and storage complexity, likely misses the cycle.
- Reuses: Core analysis only.

## Recommended Approach
Choose Approach B. It is the most ambitious option that still has a clean path to a public artifact today, and it turns Stack Sleuth into a stronger reusable engine rather than another isolated demo mode.

## Architecture
- Keep `src/portfolio.js` responsible for parsing and prioritizing labeled portfolios.
- Add `src/forge.js` as a shared engine that consumes a portfolio report, selects representative incidents, deduplicates by signature, derives suggested case labels, and produces exportable casebook text plus render helpers.
- Extend the CLI with a dedicated forge mode that reads portfolio input from `--forge <path|->` and returns text, JSON, or Markdown.
- Extend the browser workflow so portfolio-shaped input can populate new Casebook Forge cards and copy-ready export text.
- Keep the forged export aligned with the existing `=== label ===` history format so no downstream parser changes are required.

## Data Flow
1. Input enters as a labeled portfolio using `@@@ pack-label @@@` sections with embedded incident-pack sections.
2. Portfolio Radar parses and analyzes each pack to produce current digests, casebook matches, regression signals, and timeline signals.
3. Casebook Forge walks runnable packs, deduplicates current incident groups by signature, and chooses a representative trace for each forged case.
4. Each forged case receives a suggested label derived from known history labels when available, otherwise from culprit path + diagnosis tags.
5. The forge engine emits:
   - a summary object,
   - per-case metadata,
   - text and Markdown summaries,
   - ready-to-save casebook export text in existing labeled-history format.
6. CLI and browser copy flows expose the same forged export so users can paste it directly into later Stack Sleuth runs.

## Error Handling
- Empty or unlabeled portfolio input returns a clear forge-specific failure in the CLI and an empty-state explanation in the browser.
- Portfolios with no runnable packs still surface why they failed instead of silently returning an empty export.
- Cases with no matched history labels fall back to deterministic suggested labels instead of blank labels.
- Duplicate suggested labels are disambiguated with stable numeric suffixes.

## Testing
- Add forge engine tests for deduplication, label derivation, export format, and summary rendering.
- Extend CLI tests for `--forge`, JSON and Markdown modes, malformed portfolios, and empty-runnable failure paths.
- Extend browser tests for portfolio-to-forge rendering, copy behavior, and result-card reset behavior.
- Extend examples and README contract tests to document the new workflow.

## Success Criteria
- `npm test` passes in the Stack Sleuth repo.
- `node ./bin/stack-sleuth.js --forge <portfolio>` emits a reusable labeled casebook export.
- Portfolio-shaped browser input surfaces forged case summaries and a copy-ready export block.
- README positions Casebook Forge as the bridge from active triage to reusable incident memory.

## Open Questions Resolved For This Cycle
- **Should forge require new input syntax?** No. Reuse portfolio syntax.
- **Should forge persist state?** No. Export text is the artifact.
- **Should known history labels be editable this cycle?** No. Reuse closest known labels when available and otherwise derive deterministic fallback labels.
