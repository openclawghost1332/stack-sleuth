# Design: Stack Sleuth Handoff Briefing

Generated by autonomous office-hours + builder-mode scoping on 2026-04-30.
Status: DRAFT, auto-approved for cron execution after internal review.
Mode: Builder

## Problem Statement
Portfolio Radar can rank incident packs and show a response queue, but teams still need to manually turn that output into a copy-ready human handoff. The product is strong at diagnosis and decent at coordination, yet it still stops short of the moment where someone needs to paste owner-specific asks into Slack, a ticket, or a shift note.

## What Makes This Cool
Paste a labeled incident portfolio and Stack Sleuth emits deterministic handoff packets: who should look first, why each pack matters now, which remembered fixes or runbooks should be reused, and which risky packs still need fresh ownership. That feels like a real team workflow, not just another analyzer.

## Premises
1. The best next Stack Sleuth move is a higher-leverage team handoff layer, not another isolated parsing mode.
2. The existing portfolio report already contains enough signal to synthesize shareable handoff packets without adding persistence or external integrations.
3. The workflow should reuse the same labeled portfolio input and stay deterministic, copy-paste friendly, and browser-demoable.
4. Packs with no recalled owner or missing runbook context should become first-class handoff packets instead of being buried in summary text.

## Approaches Considered

### Approach A: Expand Portfolio Radar copy only
- Summary: keep the existing portfolio mode and add a few extra summary lines for handoff language.
- Effort: S
- Risk: Low
- Pros: tiny diff, low risk, no new surface.
- Cons: weak product boundary, not obviously reusable, still too manual.
- Reuses: current portfolio summary renderers only.

### Approach B: Dedicated Handoff Briefing workflow (recommended)
- Summary: add a shared handoff engine that turns a portfolio report into owner packets and gap packets, expose it through CLI, browser cards, examples, and README.
- Effort: M
- Risk: Medium
- Pros: clear outsider story, stronger coordination artifact, compounds response queue and casebook memory work.
- Cons: touches several surfaces and needs careful packet wording.
- Reuses: `analyzeIncidentPortfolio`, response queue signals, runbook recall metadata, existing browser portfolio flow.

### Approach C: Persistent incident board
- Summary: add editable assignments, status tracking, and saved packet state.
- Effort: XL
- Risk: High
- Pros: stronger long-term product.
- Cons: drags the repo into storage and app-state work that does not fit this cycle.
- Reuses: current portfolio analysis, but adds a separate application layer.

## Recommended Approach
Choose Approach B. It is the boldest one-cycle slice that outsiders will understand immediately and it turns Stack Sleuth into a real handoff tool without abandoning the local deterministic product shape.

## Accepted Scope
- add a shared Handoff Briefing engine that accepts a raw labeled portfolio or an already analyzed portfolio report
- generate owner-focused handoff packets from the response queue
- generate explicit gap packets for unowned packs and missing runbooks
- expose the handoff summary and packet export in CLI text, markdown, and JSON output
- surface the handoff summary and export in the browser portfolio flow without displacing Portfolio Radar as the primary view
- add one example portfolio and README section that make the workflow obvious to outsiders

## Deferred Scope
- persistent assignments or acknowledgements
- ticket, Slack, PagerDuty, or incident-platform integrations
- editing owners or runbooks in the browser
- heuristic ownership guesses without casebook evidence
- auto-generated remediation beyond already recalled guidance

## Architecture
1. Add `src/handoff.js` as the shared packet builder and renderer.
2. The engine should accept either a raw labeled portfolio string or an existing portfolio report.
3. Build packets from three signal groups:
   - owner packets from `responseQueue`
   - ownership gap packets from `unownedPacks`
   - runbook gap packets from `runbookGaps`
4. Each packet should preserve the main pack labels, the reason the pack is urgent now, any remembered summary/fix/runbook guidance, and a short deterministic ask.
5. Add CLI routing through `--handoff <path|->` and browser portfolio cards for the summary plus multiline export text.
6. Keep the export format plain text so it is easy to paste into chat, docs, or tickets.

## Error Handling
- `--handoff` should fail clearly when the input is not a labeled portfolio.
- `--handoff` should fail clearly when the portfolio has no runnable packs.
- If a portfolio has owned packets and gap packets, both should be emitted.
- If several packs map to the same owner, merge them into one owner packet with de-duplicated summaries, fixes, and runbooks.

## Testing Strategy
- add unit tests for owner packet aggregation, gap packet generation, deterministic ordering, and multiline export rendering
- add CLI tests for `--handoff` text, markdown, and JSON flows plus validation failures
- add browser tests for portfolio rendering, reset behavior, and example copy
- add README assertions so the new workflow stays documented
- run the full Node suite after implementation

## Success Criteria
1. `stack-sleuth --handoff <portfolio>` emits a shareable handoff summary plus copy-ready packets.
2. The handoff export clearly includes owner packets and explicit gap packets when ownership or runbook recall is missing.
3. The browser portfolio flow visibly shows the new handoff layer without changing the portfolio input contract.
4. Tests and docs cover the new workflow end to end.
