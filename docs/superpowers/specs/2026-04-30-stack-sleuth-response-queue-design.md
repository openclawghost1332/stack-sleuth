# Design: Stack Sleuth Response Queue

Generated by office-hours + CEO review on 2026-04-30

## Problem Statement
Stack Sleuth can already rank several incident packs with Portfolio Radar, but release-level triage still stops short of the team question that matters next: who should act first, which packs already have a recalled owner or runbook, and where are the dangerous unknowns with no clear routing. Right now the repo is strong at analysis and weaker at deterministic coordination.

## What Makes This Cool
Paste a labeled portfolio with historical casebook context and Stack Sleuth does not just say which pack is scary. It builds an owner-aware response queue, calls out runbook coverage, and surfaces the unowned gaps that will slow a real incident review.

## Premises
1. The next compounding move for Stack Sleuth is a higher-leverage team workflow, not another isolated parsing lens.
2. Existing Casebook Radar guidance metadata (`owner`, `runbook`, `fix`, `summary`) is rich enough to build a first deterministic response queue.
3. The feature should extend Portfolio Radar instead of inventing a separate storage system, assignment system, or hosted product.
4. Novel incidents and packs without recalled owners should be made painfully visible instead of hidden behind summary prose.

## Approaches Considered

### Approach A: Minimal viable, text-only queue add-on
- Summary: extend Portfolio Radar summary text with owner counts and runbook coverage counts, but no deeper structured queue.
- Effort: S
- Risk: Low
- Pros: quick, low integration risk, easy to test.
- Cons: not a strong enough visible leap, weak browser story, not much reusable structure.
- Reuses: existing portfolio report and rendering helpers.

### Approach B: Recommended, shared response-queue aggregation inside Portfolio Radar
- Summary: add shared owner-routing and runbook-gap aggregation derived from pack casebook matches, then expose it in CLI, markdown, JSON, browser cards, examples, and README.
- Effort: M
- Risk: Medium
- Pros: visibly deepens Stack Sleuth into a team workflow, stays deterministic, compounds existing portfolio and runbook recall work.
- Cons: touches several integration surfaces, requires careful ranking and wording.
- Reuses: casebook matching guidance, portfolio ranking, existing browser cards, existing portfolio renderers.

### Approach C: Ideal architecture but out of scope, persistent coordination board
- Summary: add saved assignments, acknowledgements, pack states, and external ticket links on top of the response queue.
- Effort: XL
- Risk: High
- Pros: stronger long-term product shape.
- Cons: needs storage, editing flows, sync, and much more UX than fits this cycle.
- Reuses: existing analysis core, but creates a parallel application layer.

## Recommendation
Choose Approach B. It is the boldest one-cycle slice that outsiders will understand immediately and it pushes Stack Sleuth from analysis toward coordination without dragging in persistence or integrations.

## Accepted Scope
- extend portfolio analysis with a structured response queue derived from casebook guidance matches
- aggregate recalled owners, runbooks, fixes, and source pack labels into queue entries
- surface unowned packs and runbook gaps explicitly
- add coverage metrics to the portfolio summary
- render the response queue and runbook gaps in text, markdown, JSON, and browser UI
- add at least one example portfolio that demonstrates owner routing plus a novel-gap case
- document the workflow in the README

## Deferred Scope
- persistent assignments or acknowledgements
- external incident or ticket integrations
- manual editing of owners inside the browser
- non-casebook ownership heuristics beyond recalled metadata
- AI-authored remediation or routing

## Architecture
1. **Portfolio aggregation layer** in `src/portfolio.js` gains queue-building helpers that derive routing data from each runnable pack report.
2. **Casebook-derived guidance extraction** reuses `report.casebook.incidents[*].matchingGuidance` to collect owners, runbooks, fixes, and summaries while preserving which pack triggered them.
3. **Structured queue model** groups packs into:
   - owner queue entries for recalled owners
   - runbook coverage entries and gaps
   - unresolved packs with no recalled owner or no recalled runbook
4. **Rendering layer** extends existing portfolio text and markdown summaries plus browser cards.
5. **CLI and browser surfaces** keep the same portfolio input format, so the workflow gets stronger without changing how users paste data.

## Data Flow
1. User supplies a labeled portfolio through CLI or browser.
2. `parseIncidentPortfolio()` splits it into labeled packs.
3. `analyzeIncidentPack()` runs per pack as it does today.
4. New portfolio helpers inspect each pack report and extract:
   - matched owners from casebook guidance
   - matched runbooks
   - matched fixes or summaries for copy-ready context
   - novel incident presence and unrunnable status
5. Aggregation produces:
   - `responseQueue`: ranked owner entries with linked packs and guidance
   - `runbookCoverage`: summary counts and per-pack gaps
   - `unownedPacks`: packs that need fresh triage
6. Renderers and browser UI present the existing priority queue plus the new coordination view.

## Ranking Rules
- Queue entries sort first by the highest-ranked pack they cover, then by novel-incident count, then by total impacted packs, then owner name.
- Unowned packs always remain separately visible even if the same portfolio also has owned packs.
- Runbook gaps prefer packs that rank earlier in the main portfolio queue.
- Duplicate owners or runbooks collapse into one entry with merged pack labels.

## Error Handling
- If a portfolio is unrunnable, the queue should stay empty and the existing omission guidance remains the primary message.
- If a pack has casebook matches but no `owner`, it should count as a routing gap rather than silently disappearing.
- If a pack has an owner but no `runbook`, it should still appear in the owner queue and also in runbook gaps.
- If multiple owners are recalled for one pack, show each owner entry and keep the pack label attached to both.

## Testing Strategy

- extend rendering tests for text and markdown queue sections

## Success Criteria
- a user can paste one labeled portfolio and immediately see who likely owns the known incidents
- the output clearly calls out missing owners or missing runbooks for risky packs
- the browser demo shows the new coordination layer without adding a new mode or input format
- the repo story reads like a real product step up, not a small cosmetic tweak

## Implementation Notes
- prefer additive changes inside Portfolio Radar over creating a separate mode
- keep JSON schema legible and deterministic
- keep copy text short enough for incident notes or PR comments
- preserve the existing CLI and browser input contract
