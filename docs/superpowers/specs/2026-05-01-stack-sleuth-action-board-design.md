# Design: Stack Sleuth Action Board

Generated by autonomous office-hours + builder-mode scoping on 2026-05-01.
Status: DRAFT, auto-approved for cron execution after internal review.
Mode: Builder

## Problem Statement
Stack Sleuth can already rank incident packs, build owner handoff packets, preserve reusable casebook datasets, and replay saved response bundles. What it still lacks is one durable coordination surface that survives the jump from live portfolio triage to saved artifacts. Today the routing gaps and concrete asks are strongest at portfolio time, then partially disappear when a team reopens the saved dataset or bundle later.

## What Makes This Cool
Paste a release-worth of incident packs, or reopen a saved dataset or response bundle, and Stack Sleuth produces a deterministic Action Board: owner lanes, routing-gap cards, runbook-gap cards, and steward backlog cards that tell a team what to route now, what documentation debt is blocking trust, and what still needs follow-up across release windows. That turns the repo from a smart analyzer into a compact incident coordination workbench.

## Premises
1. The strongest next move is to deepen the existing Stack Sleuth product spine, not spin up another isolated parser mode.
2. Saved artifacts should preserve operational asks, not just diagnosis summaries, otherwise replay loses too much leverage.
3. The board must stay deterministic and local-first, with no new storage layer or third-party integrations.
4. Backward compatibility matters: older saved datasets and response bundles must still replay clearly, even if they cannot preserve every new board lane.

## Approaches Considered

### Approach A: Expand summary copy only
- Summary: add more board-like sentences to Portfolio Radar, dataset replay, and bundle replay summaries.
- Effort: S
- Risk: Low
- Pros: tiny diff, no schema changes.
- Cons: not a real product surface, still loses concrete asks in saved artifacts, weak outsider story.
- Reuses: existing summary renderers only.

### Approach B: Dedicated Action Board engine with saved-artifact preservation (recommended)
- Summary: add a shared board engine that can build deterministic cards from live portfolio reports and replayed datasets or bundles, preserve routing and runbook gaps in saved artifacts, and surface the board in CLI, browser, docs, and examples.
- Effort: M
- Risk: Medium
- Pros: clearly visible workflow, compounds handoff plus steward work, makes replay materially more actionable.
- Cons: needs versioned dataset and bundle compatibility work plus careful browser copy.
- Reuses: `analyzeIncidentPortfolio`, `buildHandoffBriefing`, dataset replay, bundle replay, steward backlog signals.

### Approach C: Editable persistent incident board
- Summary: add saved acknowledgement state, inline editing, and durable assignments.
- Effort: XL
- Risk: High
- Pros: strongest long-term coordination product.
- Cons: drags the repo into app-state, write flows, and persistence complexity that do not fit this cycle.
- Reuses: current portfolio and replay analysis, but requires a separate application layer.

## Recommended Approach
Choose Approach B. It is the boldest one-cycle slice that is still real: one shared board model, preserved saved-artifact coordination data, and a visibly thicker workflow that outsiders can understand quickly.

## Accepted Scope
- add a shared `src/action-board.js` engine that accepts a live portfolio report, a saved Casebook Dataset replay, or a saved Response Bundle replay
- define deterministic board lanes for owner-routed work, ownership gaps, runbook gaps, and steward backlog follow-ups
- preserve routing-gap and runbook-gap data in saved datasets and response bundles so replay remains actionable
- version the saved dataset and response bundle formats forward while keeping older versions replayable
- surface Action Board summary and cards in CLI text, markdown, JSON, browser cards, examples, and README copy
- keep Portfolio Radar as the primary live analysis while letting the board sit on top as the coordination layer

## Deferred Scope
- editing cards, acknowledging completion, or persisting mutable board state
- Slack, GitHub issue, PagerDuty, or ticket-system delivery
- heuristic owner guesses without casebook evidence
- raw trace recovery from saved artifacts beyond what is already preserved
- separate standalone board-only app shell

## Architecture
1. Add `src/action-board.js` as the shared engine and renderer.
2. The engine should normalize three input families into one board model:
   - live portfolio reports, preserving owner packets plus routing and runbook gaps
   - saved dataset replays, preserving any saved gaps plus steward backlog and replay caveats
   - saved response bundle replays, delegating to the embedded dataset and bundle manifest context
3. Each board card should expose:
   - lane
   - title
   - affected labels
   - why-now reasons
   - recalled guidance when available
   - deterministic ask
   - replay note when the card was reconstructed from an older artifact
4. Extend `src/dataset.js` to preserve action-board-relevant routing data and bump the dataset schema version while keeping prior saved versions readable.
5. Extend `src/bundle.js` and `src/bundle-replay.js` so response bundles carry the richer dataset state and summarize the Action Board in replay.
6. Update `src/main.js`, `index.html`, and examples so the browser shows Action Board summary and top cards for live portfolio analysis and saved artifact replay.

## Error Handling
- `--board` should fail clearly when the input is not a labeled portfolio or saved replay artifact.
- Older datasets or bundles that lack saved routing gaps should still render a board with an explicit replay limitation note instead of pretending the data exists.
- If a portfolio has no runnable packs, the board should report that it cannot build action cards yet.
- If a saved artifact preserves only owner and steward lanes, missing routing-gap or runbook-gap lanes must be called out as unavailable, not silently omitted.

## Testing Strategy
- add unit tests for board lane assembly, deterministic ordering, replay caveats, and preserved versus reconstructed cards
- add dataset and bundle compatibility tests covering version bumps plus replay of older artifacts
- add CLI tests for Action Board text, markdown, and JSON flows on live portfolios and saved artifacts
- add browser tests for portfolio rendering, saved dataset replay, saved response bundle replay, copy behavior, and reset behavior
- update README assertions so the new workflow and saved-artifact guarantees stay documented
- run the full Node suite after implementation

## Success Criteria
1. `stack-sleuth --board <portfolio>` emits a clear Action Board summary and deterministic cards.
2. Saved Casebook Dataset and Response Bundle replays preserve enough routing and gap detail to rebuild the Action Board honestly.
3. Browser users can see the Action Board alongside Portfolio Radar and when replaying saved artifacts.
4. Older saved artifacts still replay, with explicit caveats when they cannot preserve the new board lanes.
5. Tests and docs cover the feature end to end.
