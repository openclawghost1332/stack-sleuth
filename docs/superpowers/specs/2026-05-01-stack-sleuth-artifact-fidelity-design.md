# Design: Stack Sleuth Artifact Fidelity

Generated by autonomous office-hours + CEO review on 2026-05-01.
Status: DRAFT, auto-approved for cron execution after internal review.
Mode: Builder

## Problem Statement
Stack Sleuth has a strong saved-artifact story, but two gaps weaken the public product. First, `--dataset --json` emits a partial Casebook Dataset payload because the CLI serializer drops routing gaps, runbook gaps, steward state, and Action Board state even though the in-memory dataset object has them. Second, `--board` can rebuild an Action Board from pasted response-bundle JSON, but not directly from the saved response-bundle directory that Stack Sleuth exports. Both gaps make durable incident handoff less trustworthy than the repo now claims.

## What Makes This Cool
A team can export a response bundle, send the folder to another responder, and that person can run `stack-sleuth --board ./saved-bundle` to reopen the coordination board immediately. A team can also save `stack-sleuth --dataset --json` output and know that the routing lanes, runbook gaps, steward backlog, and board summary survived the round trip. That turns saved artifacts from “mostly replayable” into a more believable portable workflow.

## Premises
1. Saved-artifact fidelity is product work, not background cleanup.
2. The best one-cycle move is to deepen Stack Sleuth’s existing artifact spine rather than branch into a new parser or demo.
3. Direct support for the exported response-bundle directory is more valuable than adding another new output mode.
4. Backward compatibility must remain explicit: older artifacts still replay, but missing preserved fields must stay visibly caveated.

## Approaches Considered

### Approach A: Fix dataset JSON fidelity only
- Summary: patch dataset serialization and stop there.
- Effort: S
- Risk: Low
- Pros: correctness-first, quick ship.
- Cons: weaker outward-facing workflow story, leaves directory handoff clumsy.

### Approach B: Saved-artifact fidelity pass with direct bundle-directory Action Board intake (recommended)
- Summary: fix dataset round-tripping and route saved response-bundle directories directly into Action Board.
- Effort: M
- Risk: Medium
- Pros: visible CLI upgrade, trust repair, compounds the response-bundle export line.
- Cons: touches shared CLI routing and serializer code.

### Approach C: Full saved-bundle ingestion across every portfolio-derived mode
- Summary: make all portfolio-derived commands accept saved bundle directories.
- Effort: L
- Risk: High
- Pros: strongest long-term platform move.
- Cons: too large for this cycle, easy to blur mode semantics.

## Recommended Approach
Choose Approach B. It fixes the honesty bug and improves the most natural saved-artifact reuse path in one coherent slice.

## Accepted Scope
- Preserve `routingGaps`, `runbookGaps`, `steward`, and `board` in Casebook Dataset JSON output.
- Keep dataset replay deterministic and backward compatible with older versions.
- Teach `--board` to accept saved response-bundle directories directly by reusing the existing response-bundle directory loader.
- Add tests proving the serializer round trip and the new CLI intake path.
- Update README documentation to show the saved-artifact workflow clearly.

## Deferred Scope
- Generic directory ingestion for every saved-artifact family.
- New browser UI surfaces beyond what preserved dataset fidelity already enables.
- New workflow modes, external integrations, or mutable coordination state.

## Architecture
1. Patch `bin/stack-sleuth.js` dataset serialization so Casebook Dataset JSON output includes the full preserved coordination payload already present in memory.
2. Keep replay normalization in `src/dataset.js` authoritative, so older datasets without these fields still rebuild with explicit caveats.
3. Reuse `readReplayBundleInput()` for `--board` when the provided path is a saved response-bundle directory, instead of treating directories as plain text files.
4. Let `src/action-board.js` continue to normalize saved bundle replay objects via the embedded `casebook-dataset.json` artifact, keeping source-kind labeling and replay notes intact.
5. Cover the new contract with focused CLI tests plus README assertions.

## Error Handling
- `--board` should still fail clearly when the input is neither a labeled portfolio nor a saved Stack Sleuth replay artifact.
- Saved response-bundle directories missing `manifest.json` or `casebook-dataset.json` should surface the existing replay-style errors instead of falling through silently.
- Older datasets that lack preserved routing or steward fields must continue to replay with explicit limitation notes.

## Testing Strategy
- Add CLI tests that save `--dataset --json`, parse the result, and assert the preserved coordination fields exist.
- Add a CLI test that points `--board` at an exported response-bundle directory and verifies a successful Action Board summary.
- Add replay coverage showing the preserved dataset JSON still reopens correctly.
- Update README assertions for the direct Action Board saved-bundle workflow and dataset-fidelity promises.
- Run the full `npm test` suite after implementation.

## Success Criteria
1. `stack-sleuth --dataset --json` emits Casebook Dataset JSON that preserves routing gaps, runbook gaps, steward state, and board state.
2. `stack-sleuth --board <saved-response-bundle-directory>` succeeds on an exported bundle directory.
3. Older saved datasets and bundles still replay, with honest limitation notes when needed.
4. Tests and README document the saved-artifact workflow end to end.
