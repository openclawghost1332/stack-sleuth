# Stack Sleuth Dataset Replay Design

Generated by autonomous builder-mode scoping on 2026-05-01.
Status: DRAFT

## Problem Statement
Stack Sleuth can already emit a portable Casebook Dataset, but saved datasets are still weak after export. They can be reused indirectly through `--history`, yet they cannot be reopened as a first-class investigation artifact, and version drift is only partially guarded.

## What Makes This Cool
One saved JSON blob becomes a portable incident handoff you can reopen later in either the CLI or browser. Instead of seeing opaque JSON, you get the useful story back: ranked packs, routing guidance, recurring hotspots, merged casebook entries, and the reusable export text that keeps incident memory moving forward.

## Premises
1. Saved datasets should become a first-class artifact before Stack Sleuth chases persistent storage.
2. Replay should stay deterministic and artifact-bound, never inventing raw trace details that were not preserved.
3. Version checks and shared renderers are product features because durable handoffs require trust.

## Approaches Considered

### Approach A: Version checks only
Summary: Tighten dataset validation on `--history` and stop there.
Effort: S
Risk: Low
Pros: Fastest safety improvement.
Cons: Leaves datasets opaque and underpowered.
Reuses: current dataset parsing path.

### Approach B: First-class dataset replay
Summary: Add shared dataset validation, replay, and rendering helpers, expose them through a CLI replay mode, and auto-detect pasted dataset JSON in the browser.
Effort: M
Risk: Medium
Pros: Strong product story, shared rendering, visible browser artifact, clear forward-safe errors.
Cons: Touches dataset engine, CLI routing, browser routing, examples, docs, and tests.
Reuses: `buildCasebookDataset`, portfolio response queue data, recurring signals, merged casebook export.

### Approach C: Persistent dataset library
Summary: Add local storage, saved dataset browsing, and editing workflows.
Effort: XL
Risk: High
Pros: Strongest long-term product direction.
Cons: Too much state and lifecycle work for one cycle.
Reuses: all dataset and portfolio flows.

## Recommended Approach
Approach B. It deepens the artifact boundary, compounds the current dataset work, and stays realistic for one aggressive cycle.

## Architecture
- Extend `src/dataset.js` so it owns dataset constants, validation, replay inspection, and shared text or markdown summary rendering.
- The new replay helper accepts a JSON string or object, validates `kind` and `version`, and returns a normalized dataset report that browser and CLI callers can trust.
- `bin/stack-sleuth.js` adds `--replay-dataset <path|->` for reopening a saved dataset without raw portfolio input.
- `--history` continues to accept saved datasets, but now rejects unsupported versions with an explicit dataset-version error.
- The browser shared textarea checks for saved dataset JSON before notebook, portfolio, or incident-pack parsing. When present, it replays the saved response queue, recurring incidents, recurring hotspots, case summaries, and merged export text into the existing cards.
- Shared dataset text and markdown renderers replace hand-written CLI strings so replay and generation stay aligned.

## Data Flow
1. Portfolio input still builds datasets through `buildCasebookDataset`.
2. Saved dataset JSON flows through `inspectDatasetInput` or a new replay helper.
3. Validation checks `kind`, `version`, summary shape, and non-empty `exportText` where needed.
4. Replay mode returns preserved artifact fields only: summary, response queue, recurring signals, cases, and export text.
5. Browser or CLI renderers display those preserved fields without assuming raw traces still exist.

## Error Handling
- Replay mode fails on unsupported `kind` with a dataset-specific error.
- Replay mode fails on unsupported `version` with a clear supported-version message.
- `--history` surfaces the same unsupported-version error before attempting casebook analysis.
- Browser dataset detection falls back to normal parsing only when the input is not dataset JSON, not when it is malformed Stack Sleuth dataset JSON.

## Testing
- Dataset unit tests cover valid replay inspection, wrong kind, unsupported version, and shared renderers.
- CLI tests cover `--replay-dataset` in text, markdown, and JSON modes plus dataset-version failures.
- Browser tests cover auto-detecting saved dataset JSON, replaying saved cards, load-example wiring, and resetting back to the empty state.
- README tests update so docs stay aligned with the replay command and dataset story.

## Success Criteria
1. `stack-sleuth --replay-dataset saved.json` prints a readable dataset replay summary.
2. `stack-sleuth --history saved.json` still works for supported datasets and fails clearly on unsupported versions.
3. Pasting a saved dataset JSON blob into the browser demo replays the saved portfolio signals without needing the original portfolio text.
4. CLI, browser, tests, and README use aligned dataset wording.

## Deferred
- Persistent dataset catalogs.
- Dataset editing or conflict-resolution workflows.
- Reconstructing support frames or culprit-level trace summaries from saved artifacts.
