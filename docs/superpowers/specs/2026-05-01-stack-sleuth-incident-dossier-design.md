# Design: Stack Sleuth incident dossier export

Generated by autonomous evolution cycle on 2026-05-01
Status: APPROVED_FOR_AUTONOMOUS_EXECUTION
Mode: Builder

## Problem Statement
Stack Sleuth can already analyze incident packs, rank labeled portfolios, build owner handoff packets, forge reusable casebooks, and preserve portable dataset state. But the output boundary is still weak. Today the user gets terminal text, markdown, JSON, or the live browser app. None of those is the right artifact for a follow-the-sun handoff, an exec status post, or a phone-friendly incident review that should stand alone after Stack Sleuth finishes running.

The next cycle should turn Stack Sleuth's strongest structured workflows into a shareable artifact, not another narrow analysis mode.

## What Makes This Cool
This gives Stack Sleuth a real output product:

```text
incident pack / portfolio / notebook / workspace / capsule
  -> existing Stack Sleuth analyzers
  -> standalone HTML dossier
  -> shareable incident command deck
```

The dossier becomes the outsider-visible artifact. It is more legible than raw JSON, more portable than a live browser session, and more compelling than terminal output alone.

## Current State
- Incident Pack Briefing and Portfolio Radar already produce dense structured reports with summaries, findings, queues, gate calls, recurring signals, and preserved export text.
- Notebook, workspace, and capsule routes already normalize into pack or portfolio workflows.
- There is no standalone HTML artifact path yet, so sharing still depends on copied text or rerunning the app.

## Premises
1. The highest-leverage next step is a shareable artifact for existing structured workflows.
2. The dossier must be downstream of the current analyzers, not a second analysis path.
3. The strongest first slice is pack and portfolio coverage plus notebook, workspace, and capsule routing that already lands on those workflows.
4. Direct trace, regression-only, timeline-only, casebook-only, and saved-artifact HTML exports are valuable follow-ons, but not necessary for a strong first dossier release.

## Approaches Considered

### Approach A: Markdown polish only
Summary: Improve markdown summaries and examples without adding a true standalone HTML artifact.

Effort: S
Risk: Low

Pros:
- Fastest path to a visible diff.
- Almost no new rendering surface.
- Easy to test.

Cons:
- Weak demo jump.
- Still not a self-contained phone-friendly artifact.
- Does not materially change the product boundary.

Reuses:
- Existing markdown renderers.

### Approach B: Standalone HTML dossier for structured workflows (recommended)
Summary: Add an HTML rendering layer that consumes existing pack or portfolio analysis results and emits one self-contained dossier page. Route notebook, workspace, and capsule inputs into the same flow.

Effort: M
Risk: Medium

Pros:
- Clear product-level upgrade.
- Strong GitHub Pages or sample-file story.
- Reuses current analyzers and preserved export text.
- Covers the richest workflows without multiplying logic.

Cons:
- Requires a new rendering module and output mode.
- Needs careful scope control so it stays curated instead of dumping every field.
- Requires documentation and committed sample output.

Reuses:
- `analyzeIncidentPack`
- `analyzeIncidentPortfolio`
- notebook, workspace, and capsule routing
- handoff, forge, dataset, and merge summary outputs that portfolio already composes

### Approach C: Browser export studio for all workflows
Summary: Add in-browser dossier download controls and universal HTML export across every workflow, including direct traces and saved-artifact replays.

Effort: L
Risk: High

Pros:
- Best eventual UX.
- Makes export discoverable in the browser demo.
- Broadest coverage.

Cons:
- Too much surface area for one cycle.
- High risk of ending with partial or inconsistent support.
- Likely to slow the strongest CLI-first artifact story.

Reuses:
- Browser app plus all analyzers.

## Recommendation
Choose **Approach B**.

It is ambitious enough to feel like a product upgrade, but still bounded enough to ship cleanly in one cycle.

## Accepted Scope

### Output mode
- Add CLI HTML output via `--html`.
- `--html` is supported for these structured workflows:
  - `--pack`
  - `--portfolio`
  - `--notebook`
  - `--workspace`
  - `--capsule`
- Unsupported workflows fail clearly when `--html` is requested.

### Pack dossier
The pack dossier should include:
- title and generated-at metadata
- sections present and analyses available
- headline summary
- top findings
- omissions
- current digest highlights when present
- casebook, regression, or timeline summaries when present
- hotspots and checklist

### Portfolio dossier
The portfolio dossier should include:
- title and generated-at metadata
- portfolio summary headline
- release gate verdict, blockers, warnings, and checklist
- priority queue
- response queue
- routing gaps and runbook gaps
- recurring incidents and recurring hotspots
- handoff export text
- reusable casebook export text
- dataset summary and merged-casebook review summary when available

### Routed sources
- Notebook, workspace, and capsule inputs should reuse existing routing.
- The dossier should reflect whether the routed result is a pack or portfolio view.
- No new routing rules are introduced this cycle.

### Sample artifact
- Commit at least one generated HTML dossier built from an existing sample portfolio.
- Link it from the README so the public story is obvious.

## Deferred
- Browser-side export buttons.
- HTML export for direct traces, digest-only flows, regression-only flows, timeline-only flows, casebook-only flows, dataset replay, shelf, and chronicle.
- PDF, screenshots, or email formatting.
- External sharing or hosted storage.

## Data Flow
```text
structured input
  -> existing normalization (optional)
  -> pack or portfolio analyzer
  -> dossier view model
  -> standalone HTML renderer
  -> saved or piped artifact
```

### Pack route
```text
pack / notebook / workspace / capsule
  -> analyzeIncidentPack(...)
  -> dossier sections
  -> standalone HTML
```

### Portfolio route
```text
portfolio / notebook / workspace / capsule
  -> analyzeIncidentPortfolio(...)
  -> handoff / forge / dataset / merge summaries
  -> dossier sections
  -> standalone HTML
```

## File Responsibilities
- `src/report.js` or equivalent: standalone dossier renderer and view-model helpers.
- `bin/stack-sleuth.js`: CLI output-mode parsing, HTML-mode routing, and clear unsupported-mode errors.
- `tests/report.test.mjs`: renderer coverage for pack and portfolio dossiers.
- `tests/cli.test.mjs`: CLI HTML output and routed workflow behavior.
- `tests/readme.test.mjs`: docs examples and sample dossier references.
- `sample/` and `README.md`: committed public demo artifact and usage docs.

## Error Handling
- `--html` with unsupported workflows should fail clearly and name the currently supported HTML workflows.
- Pack HTML mode should still fail if the pack contains no runnable analyses.
- Portfolio HTML mode should still fail if there are no labeled packs or no runnable pack results.
- Notebook, workspace, and capsule HTML mode should preserve their current routing and input-validation errors.

## Testing Strategy
1. Renderer tests prove pack dossiers include findings, omissions, hotspots, and checklist sections.
2. Renderer tests prove portfolio dossiers include gate, queues, recurring signals, and export text sections.
3. CLI tests prove `--html` works for pack and portfolio inputs.
4. CLI tests prove notebook, workspace, and capsule routes emit HTML through the same dossier path.
5. CLI tests prove unsupported HTML workflows fail with a useful message.
6. README tests lock usage examples and sample dossier references.

## Biggest Risks
1. Letting the dossier become a raw field dump instead of a curated share artifact.
2. Accidentally duplicating logic instead of consuming existing analysis results.
3. Expanding coverage to every workflow before the pack and portfolio experience feels strong.

## Success Criteria
- Stack Sleuth emits a standalone HTML dossier for pack and portfolio workflows.
- Notebook, workspace, and capsule inputs can reach the same dossier path through existing routing.
- The dossier looks meaningfully shareable, not just like raw JSON in HTML.
- A committed sample dossier and README link make the feature obvious to outsiders.
