# Design: Stack Sleuth response bundle export

Generated by autonomous evolution cycle on 2026-05-01
Status: APPROVED_FOR_AUTONOMOUS_EXECUTION
Mode: Builder

## Problem Statement
Stack Sleuth can already analyze labeled incident portfolios, prepare owner handoff packets, forge a reusable casebook, preserve a replayable dataset, and render a standalone dossier. But those surfaces still arrive as separate commands or copied sections. A responder who wants one complete handoff package still has to assemble the final artifact manually.

The next cycle should turn Stack Sleuth's strongest portfolio workflow into a reproducible response bundle, not another isolated output mode.

## What Makes This Cool
This gives Stack Sleuth a real packaging layer:

```text
portfolio / routed workspace / routed notebook / routed capsule
  -> existing Stack Sleuth analyzers
  -> response bundle contract
  -> one shareable incident command package
```

The bundle is both human-readable and machine-readable. A teammate can open the dossier, paste the handoff packet, archive the dataset, and preserve the merged casebook without rerunning several commands.

## Current State
- Portfolio Radar already produces the core ranking, gate, response queue, recurring-signal, and checklist surfaces.
- Handoff Briefing, Casebook Forge, Casebook Dataset, Casebook Merge, and the HTML dossier already exist as separate reusable outputs.
- Workspace, notebook, and capsule inputs already normalize into pack or portfolio workflows.
- There is no single deterministic export that packages those outputs together for team use.

## Premises
1. Packaging existing high-signal outputs is a higher-leverage move than adding another analyzer.
2. The bundle must be downstream of the current analyzers and renderers, not a second logic path.
3. The first release should focus on portfolio-shaped workflows, because multi-pack inputs are where handoff, reuse, and release coordination matter most.
4. The export contract should be a directory with fixed filenames plus one manifest that describes the bundle.
5. Routed notebook, workspace, and capsule support is only in scope when they normalize into portfolio mode. Pack-only routing should fail clearly in bundle mode.

## Approaches Considered

### Approach A: Workflow recipe only
Summary: Document the sequence of existing commands needed to build a package manually.

Effort: S
Risk: Low

Pros:
- Fastest path to a visible diff.
- No new filesystem behavior.
- Easy to test.

Cons:
- Still manual assembly.
- No new reusable contract.
- Weak public artifact jump.

Reuses:
- Existing commands only.

### Approach B: Deterministic response bundle export for portfolio workflows (recommended)
Summary: Add a bundle builder that consumes an analyzed portfolio and emits a fixed set of files plus a machine-readable manifest. Support direct portfolio input and routed workspace, notebook, or capsule inputs when they normalize into a portfolio.

Effort: M
Risk: Medium

Pros:
- Clear product-level upgrade.
- Strong sample artifact story.
- Reuses multiple existing engines at once.
- Creates a contract future tools can automate around.

Cons:
- Requires directory output semantics.
- Needs a clean manifest shape and deterministic file list.
- Needs explicit routing and failure messages.

Reuses:
- `analyzeIncidentPortfolio`
- `renderIncidentDossierHtml`
- `buildHandoffBriefing`
- `analyzeCasebookForge`
- `buildCasebookDataset`
- `analyzeCasebookMerge`
- notebook, workspace, and capsule routing

### Approach C: Browser incident-package studio
Summary: Add browser download flows and bundle management on top of the existing app.

Effort: L
Risk: High

Pros:
- Best eventual UX.
- Makes package export visually discoverable.
- Could grow into a richer product surface.

Cons:
- Too much surface area for one cycle.
- Pulls effort into browser plumbing instead of the core export contract.
- Higher risk of a partial, muddy release.

Reuses:
- Browser app plus the same bundle internals.

## Recommendation
Choose **Approach B**.

It compounds the current portfolio story into a real packaging layer without overreaching into browser-heavy UI work.

## Accepted Scope

### Export interface
- Add a CLI export option `--bundle <directory>`.
- `--bundle` is mutually exclusive with `--json`, `--markdown`, and `--html`.
- `--bundle` is supported for:
  - `--portfolio`
  - `--workspace` when the workspace normalizes into a portfolio
  - `--notebook` when the notebook normalizes into a portfolio
  - `--capsule` when the capsule normalizes into a portfolio
- `--bundle` fails clearly for:
  - direct trace and digest workflows
  - pack-only workflows
  - notebook, workspace, or capsule inputs that normalize into a single pack
  - portfolio-shaped inputs with zero runnable packs

### Bundle contract
The export directory contains exactly these files:
- `manifest.json`
- `incident-dossier.html`
- `portfolio-summary.md`
- `handoff.md`
- `casebook.txt`
- `casebook-dataset.json`
- `merge-review.md`

### Manifest responsibilities
`manifest.json` should include:
- bundle kind and version
- generated timestamp
- source mode and source label
- portfolio summary headline and release gate verdict
- pack counts and runnable-pack counts
- response owner count
- recurring incident and hotspot counts
- deterministic file list

### File responsibilities
- `incident-dossier.html`: standalone dossier rendered from the existing portfolio dossier path.
- `portfolio-summary.md`: markdown portfolio summary rendered from the current portfolio summary path.
- `handoff.md`: markdown handoff briefing plus copy-ready packet export.
- `casebook.txt`: reusable forged or merged casebook export text. Use the merge export because it carries living-casebook guidance forward.
- `casebook-dataset.json`: raw dataset JSON for replay.
- `merge-review.md`: markdown review summary of merge outcomes and conflicts.

### Source-of-truth rules
- The bundle must reuse existing report builders and renderers wherever possible.
- The manifest may summarize fields, but must not recompute portfolio logic independently.
- The bundle writer should operate on a small view-model object or file map generated from an analyzed portfolio.

## Deferred
- Pack-only bundle export.
- Zip or tarball packaging.
- Browser download buttons.
- Bundle replay mode.
- External storage or sharing.
- Editing or annotating the bundle after export.

## Data Flow
```text
portfolio-shaped input
  -> existing normalization (optional)
  -> analyzeIncidentPortfolio(...)
  -> bundle file map
  -> write deterministic directory
```

### Routed workflow paths
```text
notebook / workspace / capsule
  -> existing route to portfolio or pack
  -> if portfolio: response bundle
  -> if pack: fail clearly with portfolio-only guidance
```

## File Responsibilities
- `src/bundle.js`: build the response-bundle manifest and deterministic file map from a portfolio report.
- `bin/stack-sleuth.js`: parse `--bundle`, validate supported workflows, and write the bundle directory.
- `tests/bundle.test.mjs`: bundle manifest and file-map coverage.
- `tests/cli.test.mjs`: CLI bundle export and routed-workflow behavior.
- `tests/readme.test.mjs`: docs examples and committed sample bundle references.
- `sample/response-bundle/`: committed public demo artifact.
- `README.md`: bundle usage docs and sample artifact references.

## Error Handling
- `--bundle` with no supported workflow should fail clearly and name the supported bundle workflows.
- Notebook, workspace, and capsule bundle mode should explain when the routed result is a single pack instead of a portfolio.
- Bundle export should create the target directory recursively.
- If writing any bundle file fails, the CLI should exit non-zero with a clear bundle-specific error.
- Existing input-validation errors for notebook, workspace, capsule, and portfolio parsing should stay visible.

## Testing Strategy
1. Bundle builder tests prove the manifest and file map contain the expected filenames and summary fields.
2. Bundle builder tests prove dossier, markdown summary, handoff export, dataset JSON, and merge review all land in the file map.
3. CLI tests prove `--portfolio --bundle <dir>` writes the deterministic bundle files.
4. CLI tests prove notebook, workspace, and capsule routes can export a bundle when they normalize into a portfolio.
5. CLI tests prove routed pack-shaped inputs fail with a portfolio-only bundle message.
6. README tests lock the bundle docs and committed sample artifact references.

## Biggest Risks
1. Accidentally recomputing logic in the bundle layer instead of composing existing outputs.
2. Letting the bundle sprawl into a folder dump without a clear contract.
3. Expanding support to pack workflows before the portfolio-first team story is clean.

## Success Criteria
- Stack Sleuth emits a deterministic response bundle for portfolio-shaped workflows.
- The bundle contains one obvious human-facing artifact plus replay-ready and handoff-ready files.
- Routed workspace, notebook, and capsule inputs can reach the same bundle path when they normalize into a portfolio.
- A committed sample bundle and README reference make the feature obvious to outsiders.
