# Design: Stack Sleuth Response Bundle Shelf

Generated by autonomous office-hours + CEO review on 2026-05-01.

## Problem Statement
Stack Sleuth can already export one response bundle, replay one saved bundle, and compare several labeled saved bundles with Response Bundle Chronicle. The missing product layer is local persistence. Once a team accumulates several saved response bundles on disk, there is still no first-class way to treat that folder like an incident command shelf. Users must either replay bundles one by one or manually reassemble chronicle inputs.

## What Makes This Cool
A directory of saved response bundles becomes a portable incident command library. One command can catalog valid bundles, preserve broken artifacts as visible warnings, replay the latest saved command deck, surface steward and Action Board load from the newest bundle, and unlock cross-release drift from the same folder. In the browser, a responder can paste one saved shelf JSON artifact and immediately reopen that local bundle library without reconstructing inputs.

## Premises
1. Stack Sleuth's strongest compounding direction is still saved-artifact workflows, not another disconnected analyzer mode.
2. Response bundles are the richest artifact the repo exports, so giving them a first-class shelf is more valuable than another minor serializer improvement.
3. The shelf must stay deterministic and honest: preserve saved inventory and embedded coordination signals, but never imply raw-trace recovery.
4. This cycle should remain local-first and immutable. No hosted catalogs, acknowledgements, or mutable shelf state.

## Approaches Considered

### Approach A: Docs-only bundle chronicle wrapper
- Summary: tell users to concatenate saved bundles into labeled chronicle text by hand.
- Effort: S
- Risk: Low
- Pros: tiny diff, zero new artifact surface.
- Cons: not ambitious enough, still manual, weak demo value.

### Approach B: Response Bundle Shelf artifact with CLI and browser replay (recommended)
- Summary: add one portable shelf artifact plus directory-backed CLI scanning and browser replay support.
- Effort: M
- Risk: Medium
- Pros: visibly thicker product, strong reuse of existing replay and chronicle engines, obvious outsider demo.
- Cons: touches multiple integration surfaces and needs careful invalid-artifact handling.

### Approach C: Mutable multi-user bundle catalog
- Summary: layer acknowledgements, edits, and richer lifecycle state onto saved bundles.
- Effort: XL
- Risk: High
- Pros: biggest long-term product swing.
- Cons: too broad for one cycle and would drag the repo into stateful workflow semantics prematurely.

## Recommended Approach
Choose Approach B. It ships a clear local-library workflow that feels like a product jump, while staying inside one aggressive cycle.

## Accepted Scope
- Add a new portable artifact kind, `stack-sleuth-response-bundle-shelf`, version 1.
- Add a shared `src/bundle-shelf.js` module that validates saved bundle snapshots, preserves invalid warnings, summarizes the newest valid bundle, and reuses Response Bundle Chronicle when at least two valid snapshots exist.
- Add CLI `--bundle-shelf <directory>` and `--replay-bundle-shelf <path|->` workflows.
- Scan top-level response-bundle directories and top-level `.json` files in deterministic filename order, preserving invalid candidate snapshots as warning entries.
- Add browser saved-artifact detection and one loadable example for the new shelf artifact.
- Update README and tests to explain the new workflow and its saved-artifact limits.

## Deferred Scope
- Nested directory recursion.
- Cross-family shelves that mix datasets and bundles.
- Mutable coordination state, acknowledgements, or external integrations.
- New hosted preview infrastructure beyond the existing static demo.

## Architecture
1. Create `src/bundle-shelf.js` as the response-bundle sibling of `src/shelf.js`. It should:
   - validate and normalize saved shelf JSON,
   - inspect individual bundle entries with `inspectResponseBundleReplayInput()`,
   - summarize valid and invalid snapshot counts,
   - expose latest bundle facts like release gate, source workflow, steward headline, and Action Board card totals,
   - reuse `analyzeResponseBundleChronicle()` when at least two valid snapshots exist.
2. Extend `bin/stack-sleuth.js` with new option parsing, one directory scanner for top-level candidate entries, and replay-mode rendering for text, markdown, and JSON.
3. Extend browser saved-artifact detection in `src/main.js` so pasted shelf JSON replays directly and add one new example button powered by `src/examples.js` and `index.html`.
4. Keep the CLI and browser honest about saved-artifact limits. Shelf replay should only surface preserved bundle inventory and embedded dataset coordination, never trace-level recovery.

## Error Handling
- `--bundle-shelf` should fail clearly when the target path is not a directory.
- A shelf directory with zero valid bundle snapshots should exit with a bundle-shelf-specific error.
- Unsupported shelf versions and malformed saved shelf JSON should produce dedicated replay errors.
- Invalid candidate snapshots must stay visible as warning entries when valid snapshots still exist.
- Older saved response bundle versions should continue to replay through the existing bundle replay compatibility layer.

## Testing Strategy
- Add unit tests for the new shelf module, including valid snapshots, invalid warnings, chronicle reuse, replay validation, and honest rendering.
- Add CLI tests for directory scanning, replay mode, JSON and Markdown output, and invalid-path handling.
- Add browser/example coverage by extending existing example and README tests.
- Run the focused new tests first, then the full `npm test` suite.

## Success Criteria
1. `stack-sleuth --bundle-shelf <directory>` builds a deterministic saved-bundle shelf and keeps invalid bundle warnings visible.
2. `stack-sleuth --replay-bundle-shelf <path|->` replays saved shelf artifacts in text, markdown, and JSON modes.
3. Browser paste detection recognizes saved Response Bundle Shelf JSON artifacts, and the demo exposes a loadable shelf example.
4. README explains the workflow, including deterministic top-level scanning and the saved-artifact limitation note.
5. Full test suite passes.
