# Design: Stack Sleuth Workspace Fleet

Generated by autonomous office-hours + CEO review on 2026-05-01.

Mode: Builder

## Problem Statement
Stack Sleuth can already analyze one real incident workspace at a time, whether the input is a single incident room, a notebook-only folder, or a portfolio rooted at `packs/<label>/`. The missing product layer is multi-workspace triage. Once a responder has a directory full of incident rooms or release-review folders, there is still no first-class way to treat that directory like a live command queue and ask which workspace deserves attention first.

## What Makes This Cool
A directory of incident workspaces becomes a reusable command queue. One command can catalog valid and invalid workspace folders, normalize every supported workspace through the existing pack, notebook, and portfolio analyzers, rank the valid rooms by urgency, and preserve one replayable fleet artifact that reopens the triage state later without reconstructing the original rooms by hand.

## Premises
1. Stack Sleuth's strongest compounding direction is still its incident-workbench operating system, not another isolated analyzer mode.
2. Live workspace directories are the most leverage-rich next input because they meet responders at the folders they already maintain.
3. Workspace Fleet must stay deterministic and honest: preserve normalized summaries and coordination load, but never imply raw-trace recovery during replay.
4. This cycle should remain local-first and immutable. No hosted catalogs, acknowledgements, or mutable queue state.

## Approaches Considered

### Approach A: README shell loop around `--workspace`
- Summary: document how to run Stack Sleuth across several workspaces one by one.
- Effort: S
- Risk: Low
- Pros: tiny diff, no new artifact surface.
- Cons: not ambitious enough, still manual, weak demo value.

### Approach B: Workspace Fleet artifact with CLI scan and replay (recommended)
- Summary: add one portable fleet artifact plus directory-backed CLI scanning and replay support.
- Effort: M
- Risk: Medium
- Pros: visibly thicker product, strong reuse of existing workspace normalization and analyzers, obvious outsider demo.
- Cons: needs a new scoring contract and careful invalid-workspace handling.

### Approach C: mutable multi-room coordination center
- Summary: layer acknowledgements, edits, and team workflow state across workspaces.
- Effort: XL
- Risk: High
- Pros: biggest long-term product swing.
- Cons: too broad for one cycle and would drag the repo into stateful workflow semantics prematurely.

## Recommended Approach
Choose Approach B. It ships a clear command-queue workflow that feels like a product jump while staying inside one aggressive cycle.

## Accepted Scope
- Add a new portable artifact kind, `stack-sleuth-workspace-fleet`, version 1.
- Add a shared `src/workspace-fleet.js` module that validates saved fleet JSON, scores valid workspaces, preserves invalid warnings, and renders text and markdown summaries.
- Add CLI `--workspace-fleet <directory>` and `--replay-workspace-fleet <path|->` workflows.
- Scan top-level directories in deterministic filename order.
- Support single-workspace section files, notebook-only folders, and portfolio workspaces rooted at `packs/<label>/`.
- Reuse the existing workspace routing semantics so notebook-only folders normalize the same way they do in `--workspace` mode.
- Update README and tests to explain the new workflow and its replay limits.

## Deferred Scope
- Nested workspace recursion.
- Browser replay support for the new artifact family.
- HTML dossier support for fleet artifacts.
- Mutable queue state, acknowledgements, or external integrations.

## Architecture
1. Create `src/workspace-fleet.js` as the live-workspace sibling of `src/shelf.js` and `src/bundle-shelf.js`. It should:
   - inspect and validate replay artifacts,
   - normalize each workspace candidate into a routed pack or portfolio result,
   - compute a deterministic priority score plus visible reasons,
   - preserve invalid candidates as warning entries,
   - summarize top queue facts such as the top workspace, valid versus invalid counts, workspace kinds, and release-gate distribution.
2. Extract one shared workspace-routing helper from the CLI so both `--workspace` and `--workspace-fleet` can reuse the same notebook, pack, and portfolio interpretation.
3. Extend `bin/stack-sleuth.js` with new option parsing, one top-level directory scanner, replay-mode rendering for text, markdown, and JSON, and clear mode-specific failures.
4. Add one committed sample fleet artifact so docs and tests can replay the workflow without reconstructing temp directories.
5. Keep replay honest. Fleet replay should only surface preserved summaries and coordination signals, never original workspace files or trace-level detail.

## Priority Model
- Portfolio workspaces score primarily from release-gate severity and coordination burden: hold > watch > clear > needs-input, then blockers, warnings, routing gaps, runbook gaps, novel incidents, regression-new incidents, recurring incidents, recurring hotspots, and Action Board card count.
- Pack and notebook-routed workspaces score from current incident volume plus reusable evidence: casebook novel incidents, regression-new and volume-up incidents, timeline new and rising incidents, current digest group count, and hotspot concentration.
- Score ties break by workspace label in ascending order.
- The output must preserve the reasons that contributed to each score so ranking stays legible.

## Error Handling
- `--workspace-fleet` should fail clearly when the target path is not a directory.
- A fleet directory with zero valid workspaces should exit with a workspace-fleet-specific error.
- Unsupported fleet versions and malformed saved fleet JSON should produce dedicated replay errors.
- Invalid workspace candidates must stay visible as warning entries when valid workspaces still exist.
- Notebook-only workspaces must keep reusing the existing notebook normalization path; any notebook routing failure should remain visible on the candidate entry.

## Testing Strategy
- Add unit tests for the new fleet module, including mixed pack and portfolio workspaces, notebook routing, deterministic ranking, invalid warnings, replay validation, and honest rendering.
- Add CLI tests for directory scanning, replay mode, JSON and markdown output, and invalid-path handling.
- Add README coverage for the new workflow and sample replay fixture coverage if needed.
- Run focused new tests first, then the full `npm test` suite.

## Success Criteria
1. `stack-sleuth --workspace-fleet <directory>` builds a deterministic ranked queue across several top-level incident workspaces.
2. `stack-sleuth --replay-workspace-fleet <path|->` replays saved fleet artifacts in text, markdown, and JSON modes.
3. Notebook-only and portfolio workspaces keep the same routing semantics they already have in `--workspace` mode.
4. README explains the workflow, including deterministic top-level scanning and replay honesty.
5. Full test suite passes.
